# M2 — live binding + owned-tmux control

**You are building exactly two things this session:**
(a) the UI stops using mock data and renders **real sessions** streamed from the
M1 watcher; (b) Hypervisor can **spawn** new agent sessions in its own tmux
server and **send prompts** to them from the prompt bar.

Not in this session: adoption of pre-existing terminal sessions and the
opencode HTTP client (those are M2b), approval detection (M3), sqlite (M5).

## Definition of done

1. `npm run tauri dev`: the sidebar lists the same sessions as
   `cargo run --manifest-path src-tauri/Cargo.toml --bin hvscan -- --json`.
2. Run `claude -p 'count slowly to 15'` in a separate terminal: its session's
   dot turns yellow in the UI while it works and green within 2s of finishing.
3. `+ New Agent → claude code → <model>`: `tmux -L hypervisor ls` shows a new
   `hv-*` session, and the new session appears in the sidebar (discovered by
   the adapters) within ~5s.
4. Select that session, type a prompt, ⏎: within 2s the session goes yellow
   and its "last sent" line shows your text (proof the send-keys landed).
5. `python3 spike/compare.py` still prints `OK` (adapters untouched).

## Backend steps (src-tauri)

### 1. Event stream — `src-tauri/src/events.rs`

`registry::watch_sessions` already exists and is lib-accessible. In `run()`:

- `.setup(|app| { … })`: spawn a `std::thread` that calls `watch_sessions`
  with a callback which (a) attaches control tiers (below), (b) emits
  `app_handle.emit("sessions:update", &snapshot)` (import `tauri::Emitter`).
- Emit one snapshot immediately at startup, before the first fs event.
- Add `#[tauri::command] fn list_sessions()` returning the same snapshot
  (frontend calls it once on mount; events keep it fresh after).

### 2. Owned-tmux control — `src-tauri/src/control/tmux.rs`

All tmux calls target a **dedicated socket**: every command starts
`tmux -L hypervisor …`. Never touch the user's default tmux server.

- `spawn(harness, model, cwd) -> Result<String>`:
  `tmux -L hypervisor new-session -d -s hv-<8charid> -c <cwd> <cmd>` where cmd:
  - claude code → `claude --model <model>` (interactive TUI)
  - codex → `codex -m <model>`
  - cursor / opencode → return Err("not wired until M2b") — the frontend
    toasts it.
- `send(target, text)`: two calls — `send-keys -t <target> -l -- <text>`,
  then after ~150ms `send-keys -t <target> Enter`. The delay matters: TUIs
  need the literal text composed before the submit key.
- `kill(target)`: `kill-session -t <target>` (used by /kill later; expose it,
  no UI wiring required this milestone).

### 3. Correlating a spawned tmux session with its transcript

After spawning claude/codex we must learn which session file it writes, so the
sidebar row and the tmux target link up:

- Record `spawn_time` and `cwd`.
- claude code: watch `~/.claude/projects/<munged-cwd>/` (munge: `/` → `-`) for
  a new `*.jsonl` created after `spawn_time`; its basename (minus `.jsonl`) is
  the sid. codex: same idea under `~/.codex/sessions/<today>/`, file
  `rollout-*-<sid>.jsonl`.
- Poll for up to 15s (500ms interval) on the thread handling the spawn; then
  give up and leave the mapping pending (the row still appears via adapters;
  it just stays tier `observe` — log a warning).
- Persist the mapping `{ sid → tmux_session_name }` as JSON at
  `app.path().app_data_dir()/owned.json`, loaded at startup. This file is
  Hypervisor's own state — writing it does not violate the read-only rule,
  which applies to *harness* directories.

### 4. Control tiers on the snapshot

Extend the emitted session JSON with `"control"`: `"tmux"` if sid is in
owned.json, `"watch"` for cursor, else `"observe"`. (No `api` tier until M2b.)

### 5. Commands

```rust
#[tauri::command] fn list_sessions() -> Vec<SessionWire>;
#[tauri::command] fn spawn_session(harness: String, model: String, cwd: Option<String>) -> Result<String, String>;
#[tauri::command] fn send_prompt(sid: String, text: String) -> Result<(), String>;
```

`spawn_session` cwd default: the currently-selected session's cwd is passed by
the frontend; fall back to the user's home dir. `send_prompt` errors with a
clear message if sid isn't owned ("session is observe-only — adoption lands in
M2b"); frontend shows the error as a toast.

## Frontend steps (src)

### 6. Replace mock data

- On mount: `invoke('list_sessions')`, then `listen('sessions:update', …)`
  (`@tauri-apps/api` is already a dependency).
- Map wire (snake_case, see `hvscan --json`) → UI `Session` (src/types.ts):
  `harness→app`, `last_user→sent`, rust `stalled→error`, keep `title/model/
  state`, `repo/branch`, `activity` (render the activity string as the tool
  line in the transcript's now-block), `sidechains` → show `↳ n` count
  (there are no per-subagent objects in real data; `h/l` navigation simply
  no-ops when the array is empty), `control→ctl`.
- Transcript for real sessions: `you(title) … you(last_user),
  agent(last_assistant)` + activity line while working. Full transcripts are
  M5; don't fake more than the data gives.
- Delete `src/simulate.ts` and `src/mockSessions.ts` usage from the store
  (keep files deletable in one commit — remove imports, then the files).

### 7. Rewire flows

- `+ New Agent` / `/new` → `invoke('spawn_session', …)`; keep the
  harness→model picker. Toast the spawn result (or its error).
- Prompt ⏎ → `invoke('send_prompt', …)`; on error, toast the message.
- `/subagents` → same `spawn_session` with the parent's cwd, then
  `send_prompt` of `handoff — <parent title>` once the mapping resolves;
  the spawned agent appears as its own sidebar row (parent/child linking is
  a later milestone). If the mapping doesn't resolve in 15s, toast that the
  handoff prompt must be sent manually.
- Flows whose backend doesn't exist yet degrade to a named toast:
  Tab-approve → "approvals land in M3"; ⏎-adopt on observe rows → "adoption
  lands in M2b"; `/plan` `/review` `/loop` `/worktree` `/broadcast` →
  "lands in M3/M4". Do not leave any silent dead ends.

## Scope fence

- Do not modify parsing in `src-tauri/src/adapters/` (compare.py must stay OK).
- No adoption, no opencode HTTP, no approval detection, no sqlite, no
  power assertions.
- Only harness-external writes allowed: tmux commands on the `-L hypervisor`
  socket and `owned.json` in the app data dir.
- UI look stays identical to M0 — this milestone changes data, not design.
